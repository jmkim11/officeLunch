# Office Lunch Project Goal

## 목적

Office Lunch는 직장인이 제한된 점심시간 안에 회사 주변 식당을 빠르게 결정하도록 돕는 Spring Boot 백엔드 포트폴리오 프로젝트다.

이 프로젝트의 핵심은 많은 기능을 넓게 만드는 것이 아니라, 작은 MVP를 끝까지 완성해서 백엔드 엔지니어로서 다음 역량을 보여주는 것이다.

- 사용자 문제를 도메인 모델과 API로 구체화하는 능력
- Spring Boot 기반 REST API 설계와 구현 능력
- Controller-Service-Repository 계층을 역할에 맞게 분리하는 능력
- Validation과 커스텀 오류를 사용해 실패를 명확히 표현하는 능력
- 비즈니스 규칙을 테스트로 검증하는 능력
- 같은 세션의 상태 변경 정합성을 지키는 능력

## 현재 판단

추천 API 백엔드 1차 MVP를 구현했다.

- Java 21과 Spring Boot 4.1을 사용한다.
- Spring WebMVC, Validation, Actuator 의존성을 사용한다.
- `recommendation`, `restaurant` 도메인을 기준으로 패키지를 구성한다.
- 추천 기능은 Controller-Service-Repository 흐름으로 연결한다.
- 요청과 응답 DTO는 모두 일반 class로 작성한다.
- 예상 가능한 실패는 공통 커스텀 오류 응답으로 반환한다.
- 식당과 추천 세션은 교체 가능한 인메모리 Repository에 저장한다.

이번 완료 범위는 전체 Office Lunch 제품이 아니라, 사전 등록된 식당 후보를 사용하는 추천 API다. 회사 검색과 위치 기반 주변 식당 조회는 다음 제품 단계에서 별도로 설계한다.

## MVP 사용자 흐름

1. 사용자는 음식 카테고리를 선택해 사전 등록된 식당 후보로 추천 세션을 시작한다.
2. 서버는 세션 생성 응답에서 첫 식당을 추천한다.
3. 사용자가 마음에 들지 않으면 다음 식당을 추천한다.
4. 사용자가 식당을 선택하면 추천 세션이 종료된다.
5. 추천 가능한 식당이 없으면 실패 응답을 반환한다.

## MVP 범위

포트폴리오 1차 완성 범위는 다음으로 제한한다.

- 식당 데이터 모델
- 추천 세션 도메인 로직
- 추천 세션 생성 API
- 다음 식당 추천 API
- 식당 선택 API
- 예외 응답 표준화
- 도메인 테스트와 API 테스트
- 로컬 실행 가능한 Spring Boot 애플리케이션

## API

| 기능 | Method | Endpoint |
| --- | --- | --- |
| 추천 세션 생성과 첫 추천 | `POST` | `/api/recommendations/sessions` |
| 다음 식당 추천 | `POST` | `/api/recommendations/sessions/{sessionId}/next` |
| 추천 식당 선택 | `POST` | `/api/recommendations/sessions/{sessionId}/selection` |

## 제외 범위

다음 기능은 1차 MVP 이후로 미룬다.

- 실제 지도 API 연동
- 회사 검색과 회사 위치 선택
- 위치 기반 주변 식당 및 검색 반경 조회
- 외부 지도 길찾기 URL 생성
- DB, JPA, Flyway 영속화
- 사용자 로그인
- 팀 단위 취향 분석
- 리뷰 크롤링
- 복잡한 추천 알고리즘
- 관리자 페이지
- 프론트엔드
- 인프라와 배포

## Repository 원칙

현재 MVP는 외부 인프라 없이 실행할 수 있도록 인메모리 Repository를 사용한다.

- Service는 Repository 인터페이스에만 의존한다.
- 식당 조회 구현과 추천 세션 저장 구현을 분리한다.
- DB가 필요해지는 시점에 구현체를 교체한다.
- 현재 범위에서 사용하지 않는 JPA와 Flyway 코드는 미리 추가하지 않는다.

## Restaurant 모델 원칙

`Restaurant`는 식당 자체의 속성만 가진다.

포함한다.

- 식당명
- 음식 카테고리
- 주소
- 좌표
- 도보 시간
- 평균 가격
- 대기 위험도
- 운영 상태
- 외부 장소 ID

포함하지 않는다.

- 사용자의 즐겨찾기 여부
- 특정 사용자의 최근 방문일
- 추천 횟수
- 개인 평점
- 추천 세션 안에서의 선택 여부

사용자나 추천 세션에 따라 달라지는 값은 별도 모델로 분리한다.

## 품질 기준

기능 하나를 완료했다고 판단하는 기준은 다음과 같다.

- 핵심 성공 케이스가 동작한다.
- 실패 케이스가 명시적인 예외 또는 오류 응답으로 처리된다.
- 테스트가 비즈니스 규칙을 설명한다.
- API 요청/응답이 문서 또는 테스트로 확인 가능하다.
- 민감 정보가 코드와 설정 파일에 하드코딩되지 않는다.

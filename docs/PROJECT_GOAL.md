# Office Lunch Project Goal

## 목적

Office Lunch는 직장인이 제한된 점심시간 안에 회사 주변 식당을 빠르게 결정하도록 돕는 Spring Boot 백엔드 포트폴리오 프로젝트다.

이 프로젝트의 핵심은 많은 기능을 넓게 만드는 것이 아니라, 작은 MVP를 끝까지 완성해서 백엔드 엔지니어로서 다음 역량을 보여주는 것이다.

- 사용자 문제를 도메인 모델과 API로 구체화하는 능력
- Spring Boot 기반 REST API 설계와 구현 능력
- JPA, Flyway, Validation을 사용한 데이터 모델링 능력
- 비즈니스 규칙을 테스트로 검증하는 능력
- 예외 처리, 입력 검증, 운영 관측을 고려하는 기본기

## 현재 판단

이 저장소는 Spring Boot 프로젝트가 맞다.

- `org.springframework.boot` Gradle 플러그인을 사용한다.
- `OfficeLunchApplication`에 `@SpringBootApplication`이 있다.
- WebMVC, JPA, Validation, Flyway, Actuator 의존성이 포함되어 있다.

다만 현재 작성 중인 코드는 아직 컨트롤러, 서비스, JPA 매핑이 붙기 전의 도메인 초안이므로 일반 Java 프로젝트처럼 보인다. 포트폴리오 관점에서는 이 순서가 틀린 것은 아니지만, 너무 오래 순수 Java 도메인에 머무르면 Spring Boot 프로젝트로 보이지 않는다.

따라서 방향은 다음과 같다.

1. 추천 도메인 규칙을 작게 완성한다.
2. 바로 REST API로 연결한다.
3. 이후 JPA와 Flyway로 영속화한다.
4. 테스트와 문서로 구현 의도를 증명한다.

## MVP 사용자 흐름

1. 사용자는 회사 기준 점심 추천 세션을 시작한다.
2. 서버는 후보 식당 목록에서 하나의 식당을 추천한다.
3. 사용자가 마음에 들지 않으면 다음 식당을 추천한다.
4. 사용자가 식당을 선택하면 추천 세션이 종료된다.
5. 추천 가능한 식당이 없으면 실패 응답을 반환한다.

## MVP 범위

포트폴리오 1차 완성 범위는 다음으로 제한한다.

- 식당 데이터 모델
- 추천 세션 도메인 로직
- 추천 세션 생성 API
- 다음 식당 추천 API
- 식당 선택 API
- 예외 응답 표준화
- 도메인 테스트와 API 테스트
- 로컬 실행 가능한 Spring Boot 애플리케이션

## 제외 범위

다음 기능은 1차 MVP 이후로 미룬다.

- 실제 지도 API 연동
- 사용자 로그인
- 팀 단위 취향 분석
- 리뷰 크롤링
- 복잡한 추천 알고리즘
- 관리자 페이지
- 프론트엔드

## Restaurant 모델 원칙

`Restaurant`는 식당 자체의 속성만 가진다.

포함한다.

- 식당명
- 음식 카테고리
- 주소
- 좌표
- 도보 시간
- 평균 가격
- 대기 위험도
- 운영 상태
- 외부 장소 ID

포함하지 않는다.

- 사용자의 즐겨찾기 여부
- 특정 사용자의 최근 방문일
- 추천 횟수
- 개인 평점
- 추천 세션 안에서의 선택 여부

사용자나 추천 세션에 따라 달라지는 값은 별도 모델로 분리한다.

## 품질 기준

기능 하나를 완료했다고 판단하는 기준은 다음과 같다.

- 핵심 성공 케이스가 동작한다.
- 실패 케이스가 명시적인 예외 또는 오류 응답으로 처리된다.
- 테스트가 비즈니스 규칙을 설명한다.
- API 요청/응답이 문서 또는 테스트로 확인 가능하다.
- 민감 정보가 코드와 설정 파일에 하드코딩되지 않는다.

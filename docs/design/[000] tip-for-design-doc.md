# Design Doc 작성 팁

Design Doc은 길게 쓰는 문서가 아니라, 구현 전에 애매함을 줄이는 문서이다.
좋은 Design Doc은 읽는 사람이 "무엇을 만들고, 무엇을 만들지 않고, 어떤 기준으로 완료를 판단하는지" 바로 이해할 수 있어야 한다.

## 1. 먼저 문제를 좁힌다

처음부터 기술을 정하지 말고, 이 기능이 해결하는 문제를 한 문장으로 적는다.

좋은 예시는 다음과 같다.

```text
사용자가 회사 주변 식당을 추천받으려면, 먼저 회사 위치를 좌표로 확정할 수 있어야 한다.
```

아쉬운 예시는 다음과 같다.

```text
외부 API를 호출한다.
```

외부 API 호출은 해결책이지 문제 자체가 아니다.
문제를 먼저 좁히면 구현 중에 기능이 커지는 것을 막을 수 있다.

## 2. 범위에는 "하지 않을 것"도 반드시 쓴다

1인 개발에서는 할 일을 정하는 것보다 하지 않을 일을 정하는 것이 더 중요하다.
범위 밖 항목을 적어두면 MVP가 불필요하게 커지는 것을 막을 수 있다.

예를 들어 회사 검색 기능에서는 다음을 제외할 수 있다.

- 로그인 사용자별 회사 저장
- 현재 위치 기반 검색
- 자체 지도 화면
- 검색 결과 랭킹 고도화

이렇게 적어두면 나중에 새로운 아이디어가 떠올라도 MVP의 중심을 유지할 수 있다.

## 3. API는 프론트엔드 입장에서 상상해서 쓴다

백엔드 API 문서는 내부 처리보다 클라이언트가 무엇을 받을 수 있는지를 먼저 보여주는 것이 좋다.

항상 다음 세 가지를 포함한다.

- 요청 URL과 파라미터
- 정상 응답 예시
- 실패 응답 예시

응답 예시가 없으면 구현할 때마다 필드명이 흔들리기 쉽다.
필드명이 흔들리면 프론트엔드, 테스트, 문서가 함께 흔들린다.

## 4. 저장부터 생각하지 않는다

초기 MVP에서는 무조건 DB 저장부터 고민하지 않아도 된다.
특히 로그인 기능이 없다면, 사용자별 상태 저장은 생각보다 복잡해진다.

먼저 다음 질문을 한다.

```text
이 값은 꼭 서버에 저장되어야 하는가?
아니면 다음 요청으로 전달해도 되는가?
```

회사 위치 검색의 경우, 검색 결과를 바로 DB에 저장하기보다 선택된 회사의 `latitude`, `longitude`를 다음 주변 식당 조회 요청에 넘기는 방식이 더 단순하다.

## 5. 외부 API는 Adapter 뒤로 숨긴다

외부 API를 Service 코드에서 직접 호출하면 테스트가 어려워지고, API 제공자를 바꾸기 어려워진다.

다음 구조를 기본값으로 생각한다.

```text
Controller -> Service -> Port -> Adapter -> External API
```

이 구조의 장점은 다음과 같다.

- Service는 비즈니스 규칙에 집중할 수 있다.
- Adapter는 외부 API 호출과 응답 변환에 집중할 수 있다.
- 테스트에서는 실제 외부 API 대신 Fake 또는 Mock Adapter를 사용할 수 있다.
- Kakao에서 Naver로 바꾸더라도 Service 변경을 줄일 수 있다.

## 6. 예외 상황은 사용자 입력, 외부 의존성, 데이터 품질로 나눠서 찾는다

예외 케이스를 찾을 때는 막연히 생각하지 말고 세 가지 축으로 나눈다.

사용자 입력:

- 빈 검색어
- 너무 짧은 검색어
- 앞뒤 공백
- 특수문자만 있는 검색어

외부 의존성:

- 외부 API timeout
- 인증 실패
- 응답 형식 변경
- 일시적 장애

데이터 품질:

- 좌표가 없는 장소
- 주소가 없는 장소
- 이름이 중복되는 장소
- 회사가 아닌 장소가 검색되는 경우

이렇게 나누면 놓치는 케이스가 줄어든다.

## 7. Acceptance Criteria는 테스트 문장처럼 쓴다

Acceptance Criteria는 감상적인 목표가 아니라 검증 가능한 문장이어야 한다.

좋은 예시는 다음과 같다.

```text
검색어가 1자 이하이면 400 Bad Request와 INVALID_QUERY를 반환한다.
```

아쉬운 예시는 다음과 같다.

```text
검색이 잘 되어야 한다.
```

"잘"이라는 표현이 들어가면 보통 아직 기준이 부족한 것이다.
테스트 코드로 바꿀 수 있는 문장이 좋은 기준이다.

## 8. 보안은 작은 기능에도 포함한다

외부 API를 사용하는 기능에는 항상 API Key 관리 기준을 적는다.

최소 기준은 다음과 같다.

- API Key는 환경변수로 관리한다.
- API Key를 Git에 커밋하지 않는다.
- API Key를 프론트엔드에 노출하지 않는다.
- 외부 API 요청/응답 전체를 로그에 남기지 않는다.

포트폴리오에서는 이런 작은 보안 기준이 실무 감각을 보여준다.

## 9. 문서의 깊이는 기능 위험도에 맞춘다

모든 기능을 길게 쓸 필요는 없다.
하지만 다음 조건에 해당하면 자세히 쓰는 것이 좋다.

- 외부 API를 사용한다.
- DB 저장이 필요하다.
- 여러 사용자의 요청이 동시에 들어올 수 있다.
- 결제, 인증, 권한, 개인정보가 관련된다.
- 실패했을 때 사용자 흐름 전체가 막힌다.

회사 위치 검색은 외부 API, 사용자 입력, 추천 정확도와 연결되므로 비교적 자세히 쓰는 편이 좋다.

## 10. 구현 전에 마지막으로 확인할 질문

Design Doc을 다 쓴 뒤 아래 질문에 답할 수 있으면 구현을 시작해도 된다.

- 이 기능의 입력과 출력이 명확한가?
- 정상 흐름과 실패 흐름이 모두 있는가?
- 프론트엔드가 이 응답만 보고 화면을 만들 수 있는가?
- 외부 API 없이 테스트할 수 있는가?
- API Key나 민감 정보가 노출되지 않는가?
- 지금 MVP에서 하지 않을 일이 명확한가?

이 질문에 답이 막히는 부분이 있으면, 그 부분이 설계를 더 구체화해야 하는 지점이다.

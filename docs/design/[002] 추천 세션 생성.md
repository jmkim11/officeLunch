# [002] 추천 세션 생성

## 1. 목적

사용자가 음식 카테고리를 선택하면 추천 세션을 만들고 첫 번째 식당을 바로 반환한다. 클라이언트가 세션 생성 후 별도의 첫 추천 요청을 보내지 않게 해 사용자 흐름을 단순하게 유지한다.

## 2. 구현 범위

- 음식 카테고리 입력 검증
- 선택한 카테고리의 운영 중 식당 조회
- 추천 후보 순서를 세션 생성 시 한 번 섞음
- UUID 기반 추천 세션 생성
- 첫 식당 추천과 세션 저장
- Controller, Service, Repository 구성
- 모든 DTO를 일반 class로 작성

## 3. 제외 범위

- 실제 지도 또는 장소 API 연동
- 회사 위치와 검색 반경 계산
- 데이터베이스 영속화
- 사용자 로그인과 개인화
- 인프라와 배포

이번 MVP의 Repository는 메모리에 데이터를 보관한다. Repository 인터페이스를 분리해 이후 JPA 구현으로 교체할 수 있지만, 현재 기능에 필요하지 않은 영속화 코드는 추가하지 않는다.

## 4. API

```http
POST /api/recommendations/sessions
Content-Type: application/json
```

```json
{
  "category": "KOREAN"
}
```

정상 응답은 `201 Created`다.

```json
{
  "sessionId": "UUID",
  "status": "IN_PROGRESS",
  "restaurant": {
    "id": 1,
    "name": "직장인 김치찌개",
    "category": "KOREAN",
    "address": "서울특별시 강남구 테헤란로",
    "latitude": 37.5001,
    "longitude": 127.0301,
    "walkingMinutes": 5,
    "averagePrice": 10000,
    "waitRisk": "LOW",
    "externalPlaceId": "sample:1"
  }
}
```

## 5. 처리 흐름

1. Controller가 요청 DTO를 검증한다.
2. Service가 `RestaurantRepository`에서 카테고리와 운영 상태가 맞는 식당을 조회한다.
3. 후보 순서를 한 번 섞어 세션마다 첫 추천이 달라질 수 있게 한다.
4. `RecommendationSession`이 첫 추천을 기록한다.
5. `RecommendationSessionRepository`에 세션을 저장한다.
6. 세션 ID, 상태, 첫 추천 식당을 응답한다.

## 6. 비즈니스 규칙

- 음식 카테고리는 필수다.
- `ACTIVE` 상태의 식당만 후보가 된다.
- 후보가 없으면 세션을 만들지 않는다.
- 생성된 세션의 상태는 `IN_PROGRESS`다.
- 생성 응답의 식당은 이미 추천된 식당으로 기록된다.

## 7. 테스트 시나리오

- 선택한 카테고리의 운영 중 식당만 조회한다.
- 세션 저장 후 ID로 다시 조회한다.
- 세션 생성 시 첫 추천과 `IN_PROGRESS` 상태를 반환한다.
- 후보가 없으면 커스텀 오류를 반환한다.
- 정상 API 요청은 201을 반환한다.
- 카테고리가 없으면 표준 400 오류를 반환한다.
